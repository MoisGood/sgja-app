🏛️ Decisiones de Arquitectura (Discutidas)

Detección de sesiones entre navegadores/dispositivos

EnfoqueCostoProsContras
Tabla Supabase directa
upsert + SELECT cada 30s		 $0 Simple, 0 infraestructura, funcional Cada ~30s hace 1 write + 1 read (100 usuarios activos ~13 ops/s)
Supabase Realtime
Canal WebSocket por usuario		 $0 Notificación inmediata, elegante Hasta 100 canales simultáneos en plan free
Edge Function + DB Limitado Puede cerrar sesiones con service_role Heartbeat cada 30s × 100 usuarios = excede 500k/mes free
fly.io (WebSockets) $ Control total Servidor extra, mantenimiento, costo

Cuándo usar Edge Functions (resuelto)

  • ✅ **Soporte** — cerrar sesión de un usuario específico (uso ocasional, no supera límite)
  • ✅ **Envío de correos** — `/api/send-email` server-side con credenciales de `email_config`
  • ✅ **Auditoría** — registrar acciones con UID real, bypass RLS
  • ✅ **Webhooks** — notificar APIs externas (ministerio, etc.)
  • ❌ **No usar para heartbeat/ping de sesiones** — muy caro en cuota

├─ ✅ Sí, solo mostrar aviso → Tabla Supabase (upsert+SELECT, $0)

⚠️ Riesgos Identificados

  • Alto Sin paginación en 3 mantenedores → Supabase puede llegar al límite de 50k rows
  • Alto Sin RLS en tablas → cualquier request autenticado accede a todas las tablas
  • Medio Dashboard Profesor: 8 queries simultáneas → posible lentitud
  • Medio 2FA implementado pero falta probar recovery (códigos de respaldo)
  • Bajo Sin monitoreo (Vercel Analytics, Sentry) → ciegos ante errores reales
