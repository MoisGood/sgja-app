📋 Módulo: Reserva de Laboratorios

Pauta de especificaciones para implementación futura · 23 mayo 2026

1. Funcionalidad General

  • Docente selecciona laboratorio → bloques disponibles → confirma reserva
  • Otros docentes visualizan reservas ocupadas en el sistema y en un calendario compartido de Google (solo lectura)
  • Docente puede anular su reserva hasta 1 hora antes
  • Docente recibe correo de confirmación con enlace para confirmar asistencia
  • Sistema muestra mensaje explícito: el docente confirma sí o sí el uso del laboratorio
  • Solo el docente titular y admin pueden anular una reserva
  • Correo de confirmación llega al docente con copia a otros usuarios (encargado lab, UTP, etc.)

2. Validaciones del Sistema

ValidaciónDescripción
DisponibilidadMismo lab + mismo bloque + misma fecha → rechazar
Antelación mínimaNo permitir reservas con menos de 2h de anticipación
Capacidad vs cursoSi lab tiene 20 PCs y el curso tiene 40 alumnas, rechazar
Curso ya tiene claseEl curso no puede estar en dos salas simultáneas
Docente ocupadoEl profe no puede reservar si tiene horas lectivas con otro curso
Licencia/permiso/ausenteSi el docente tiene licencia, permiso o está ausente ese día → rechazar
Feriados / no laboralesBloquear días sin clases (feriados, adm., vacaciones)
Limpieza entre bloques15-30 min de bloqueo entre reservas si aplica

6. Reglas de Negocio

  • Reserva recurrente: "Todas las semanas, mismo bloque, por N semanas" — opción en formulario
  • Notificación a encargado: Al crear/cancelar reserva, notificar al encargado del laboratorio (rol Técnico)
  • Bloqueo por no-shows: Si acumula X no-shows (ej: 3), pierde privilegio de reservar por N días
  • Período máximo: No se puede reservar con más de 1 semana de anticipación (ajustable)
  • Antelación mínima: 2 horas antes del bloque (ajustable)
  • Solo lectura en calendario: Los docentes visualizan el calendario compartido de Google, no lo editan
  • Encargado lab: Rol existente o nuevo que recibe notificaciones de todas las reservas

Anexo A: App Nativa Satélite para Laboratorios

Aplicación de escritorio instalada en cada PC del laboratorio. Bloquea el equipo, autentica al usuario y registra la actividad. Se comunica con la plataforma web existente.

🎯 Propósito

  • Saber quién usó cada computador: usuario, RUT, fecha, hora inicio/fin, curso, docente a cargo
  • El PC no se puede usar sin autenticarse en la app
  • Los registros se sincronizan con la web (SQLite local como buffer offline)
  • Admin puede desbloquear cualquier PC remotamente
  • Encargado de laboratorio puede registrar usuarios in-situ si no están en el sistema

2. Flujo de Autenticación

#PasoDetalle
1 Pantalla bloqueada App inicia en pantalla completa. Muestra: "Laboratorio X — Identifíquese". Impide cualquier uso del PC
2a Iniciar con Google Abre webview con OAuth de Google (mismo que la intranet). Busca RUT en funcionarios / estudiantes. Si existe → acceso
2b Buscar por RUT Usuario escribe RUT (con/sin puntos). App consulta API o cache SQLite local. Si existe → acceso
2c Buscar por nombre Escribe nombre. API devuelve coincidencias. Selecciona → acceso
2d Usuario no encontrado Encargado lab presiona "Registrar usuario". Formulario: RUT, nombre, curso. Se guarda en usuarios_lab_temp y se da acceso inmediato
3 Acceso concedido App desbloquea pantalla. Inicia registro de sesión en SQLite. Muestra panel informativo (tiempo restante, botón cerrar sesión)
4 Cerrar sesión Usuario hace clic en "Finalizar" o el tiempo expira. App registra fecha_fin y bloquea pantalla nuevamente

8. Sincronización Offline

  1. Cada sesión se guarda en SQLite local con sincronizado = 0
  2. Cada 60 segundos (o al cerrar sesión), la app intenta POST /api/lab/sync con un lote
  3. Si no hay conexión, los registros quedan en cola local
  4. Cuando la conexión regresa, envía todos los pendientes (ordenados por fecha)
  5. El servidor responde con los IDs aceptados; la app marca esos registros como sincronizado = 1
  6. Registros con más de 30 días sin sincronizar se marcan como "perdidos"
